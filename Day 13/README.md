# Floor Notes 📒

A Flutter notes app demonstrating local persistence with the [Floor ORM](https://pub.dev/packages/floor) — a type-safe, annotation-driven database layer built on top of sqflite. The project is a practical comparison between the Floor ORM pattern and raw sqflite.

## Features

- ✅ Reactive list — UI updates automatically via `Stream` (no manual refresh)
- ✅ Persistent local storage using Floor / SQLite
- ✅ Material 3 UI with teal colour theme
- ✅ In-memory repository for widget tests (no database required)

## Architecture

```
main.dart
  └── builds AppDatabase (Floor)
  └── creates FloorNotesRepository
  └── seeds default notebook
  └── runApp(MyApp)

lib/
├── app.dart                        ← UI: home page, note list, editor sheet
└── data/
    ├── notes_repository.dart       ← Abstract interface
    ├── floor_notes_repository.dart ← Floor-backed implementation
    ├── in_memory_notes_repository.dart ← In-memory impl (for tests)
    ├── app_database.dart           ← @Database class
    ├── app_database.g.dart         ← ⚙️  Generated — do not edit
    ├── note_entity.dart            ← @Entity (notes table)
    ├── note_dao.dart               ← @dao (CRUD + Stream queries)
    ├── notebook_entity.dart        ← @Entity (notebooks table)
    └── notebook_dao.dart           ← @dao (insert + lookup)
```

The UI depends only on the `NotesRepository` interface, making it trivial to swap implementations — including the `InMemoryNotesRepository` used in widget tests.

## Database Schema

### `notebooks` table

| Column       | Type    | Constraint  | Notes                    |
|--------------|---------|-------------|--------------------------|
| `id`         | INTEGER | PRIMARY KEY | Manually assigned        |
| `name`       | TEXT    | NOT NULL    | Notebook display name    |
| `created_at` | INTEGER | NOT NULL    | Unix epoch milliseconds  |

### `notes` table

| Column        | Type    | Constraint                    | Notes                            |
|---------------|---------|-------------------------------|----------------------------------|
| `id`          | INTEGER | PRIMARY KEY AUTOINCREMENT     | Auto-generated by Floor          |
| `notebook_id` | INTEGER | FOREIGN KEY → notebooks(id)   | ON DELETE CASCADE; indexed       |
| `title`       | TEXT    | NOT NULL                      |                                  |
| `content`     | TEXT    | NOT NULL                      |                                  |
| `created_at`  | INTEGER | NOT NULL                      | Set once at creation             |
| `updated_at`  | INTEGER | NOT NULL                      | Updated on every save            |

> Foreign key cascade means deleting a notebook automatically deletes all its notes.

## Getting Started

### Prerequisites

- Flutter SDK `>=3.11.5` — [install guide](https://docs.flutter.dev/get-started/install)
- Dart SDK `>=3.11.5` (bundled with Flutter)

### 1. Clone and install dependencies

```bash
git clone <repo-url>
cd room_database_app
flutter pub get
```

### 2. Generate the Floor code

Floor uses `build_runner` to generate the concrete database implementation. This step is required before the app will compile:

```bash
flutter pub run build_runner build --delete-conflicting-outputs
```

### 3. Run the app

```bash
flutter run
```

## Code Generation

Floor's annotation processor generates `app_database.g.dart` from your annotated classes. You must re-run the generator after any change to:

- An `@Entity` class
- A `@dao` interface
- The `@Database` class

**One-off build:**
```bash
flutter pub run build_runner build --delete-conflicting-outputs
```

**Watch mode (rebuilds on save):**
```bash
flutter pub run build_runner watch --delete-conflicting-outputs
```

> ⚠️ Never edit `app_database.g.dart` manually — your changes will be overwritten on the next build.

## Key Concepts

### `@Entity` — mapping a class to a table

```dart
@Entity(
  tableName: 'notes',
  foreignKeys: [
    ForeignKey(
      childColumns: ['notebook_id'],
      parentColumns: ['id'],
      entity: NotebookEntity,
      onDelete: ForeignKeyAction.cascade,
    ),
  ],
  indices: [Index(value: ['notebook_id'])],
)
class NoteEntity {
  @PrimaryKey(autoGenerate: true)
  final int? id;           // null → new insert; non-null → update

  @ColumnInfo(name: 'notebook_id')
  final int notebookId;

  final String title;
  final String content;

  @ColumnInfo(name: 'created_at')
  final int createdAtMillis;

  @ColumnInfo(name: 'updated_at')
  final int updatedAtMillis;
}
```

### `@dao` — type-safe query methods

```dart
@dao
abstract class NoteDao {
  // Reactive query — emits a new list whenever the table changes
  @Query('SELECT * FROM notes WHERE notebook_id = :notebookId ORDER BY updated_at DESC')
  Stream<List<NoteEntity>> watchNotesForNotebook(int notebookId);

  @Query('SELECT * FROM notes WHERE id = :id LIMIT 1')
  Future<NoteEntity?> findNoteById(int id);

  @Insert(onConflict: OnConflictStrategy.replace)
  Future<int> insertNote(NoteEntity note);

  @Update()
  Future<int> updateNote(NoteEntity note);

  @Query('DELETE FROM notes WHERE id = :id')
  Future<void> deleteNoteById(int id);
}
```

### `@Database` — tying it all together

```dart
@Database(version: 1, entities: [NotebookEntity, NoteEntity])
abstract class AppDatabase extends FloorDatabase {
  NotebookDao get notebookDao;
  NoteDao get noteDao;
}
```

### Building the database in `main.dart`

```dart
Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();

  final database = await $FloorAppDatabase
      .databaseBuilder('room_database.db')
      .build();

  final repository = FloorNotesRepository(database);
  await repository.seed(); // creates the default notebook on first launch

  runApp(MyApp(repository: repository));
}
```

### Reactive UI with `StreamBuilder`

Because `NoteDao.watchNotesForNotebook` returns a `Stream`, the list refreshes automatically after any insert, update, or delete — no `setState()` or manual reload needed:

```dart
StreamBuilder<List<NoteEntity>>(
  stream: widget.repository.watchNotes(),
  builder: (context, snapshot) {
    final notes = snapshot.data ?? const <NoteEntity>[];
    // rebuild the list...
  },
)
```

## Floor vs sqflite

| Aspect              | Floor (ORM)                                              | Raw sqflite                                     |
|---------------------|----------------------------------------------------------|-------------------------------------------------|
| Setup               | Annotations + build_runner (more upfront work)           | Just add sqflite and write SQL                  |
| Type safety         | Compile-time — generated code is fully typed             | Runtime — manual `Map<String,Object?>` mapping  |
| Query style         | `@Query` annotations; SQL validated at build time        | Raw SQL strings passed directly                 |
| Boilerplate         | Less — DAO implementations are generated                 | More — manual `toMap()` / `fromMap()` required  |
| Reactive queries    | Built-in `Stream<List<T>>` support                       | Manual polling or custom stream wrappers        |
| Migrations          | `Migration` objects tracked by version                   | Manual `onCreate` / `onUpgrade` callbacks       |
| Foreign keys        | Declarative `ForeignKey` annotation                      | Manual SQL + `PRAGMA foreign_keys = ON`         |
| Performance         | Equivalent — Floor is a thin sqflite wrapper             | Marginally faster for trivial single-table apps |
| Learning curve      | Steeper (ORM concepts, build_runner)                     | Gentler (SQL knowledge required)                |
| **Best for**        | Complex apps, teams, type-safety focus                   | Simple apps, prototypes, full SQL control       |

**Use Floor when:**
- Your schema has multiple related entities
- You want reactive queries with zero boilerplate
- Type safety and self-documenting code matter
- The schema will evolve and needs tracked migrations

**Use raw sqflite when:**
- You have a single simple table
- You want full SQL control
- You need to avoid a code-generation build step
